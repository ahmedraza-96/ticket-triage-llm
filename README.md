# ticket-triage-llm

**Fine-tune a 4B open LLM to triage support tickets — match a 70B API model at
a fraction of the cost, on free T4 GPUs, with auditable before/after evals.**

[Model on HF Hub](https://huggingface.co/ahmedraza-96/Qwen3-4B-Instruct-2507-ticket-triage) ·
[LoRA adapter](https://huggingface.co/ahmedraza-96/Qwen3-4B-Instruct-2507-ticket-triage-LoRA) ·
Sibling project: [support-ticket-classifier](https://github.com/ahmedraza-96/support-ticket-classifier) (DistilBERT, same test set)

A raw customer ticket goes in; strict single-line JSON comes out:

```json
{"category": "REFUND", "reply": "I'm sorry to hear that! To get your refund started, please share your {{Order Number}}..."}
```

- **Routing + drafting in one call** — 11 intent categories plus a
  customer-facing reply, unlike the classification-only DistilBERT sibling.
- **Format learned, not forced** — JSON validity is measured with **no guided
  decoding**; the fine-tune internalizes the schema.
- **Policy safety by construction** — unknown specifics (order numbers, links)
  are emitted as `{{Placeholder}}` slots, never hallucinated; the LLM judge
  scores this explicitly.
- **Same held-out test set** as the DistilBERT project — re-derived with a
  split-parity assertion, not by copy-paste.

## How it works

| Stage | Where it runs | Entry point |
|---|---|---|
| Data prep (split-parity vs sibling repo) | local CPU | `data/prepare_data.py` |
| SFT dataset (8k stratified rows → chat JSONL) | local CPU | `data/build_sft_dataset.py` |
| QLoRA training (Unsloth, ~50 min) | Kaggle **T4** | `train/train_qlora.py` |
| Merge fp16 + publish to HF Hub | Kaggle T4 | `train/merge_and_push.py` |
| Serving | Kaggle T4, **vLLM** fp16 | `infer/serve_vllm.sh` |
| Completion artifacts (base + tuned arms) | Kaggle T4 | `infer/generate_completions.py` |
| Reference arm: Llama-3.3-70B few-shot | local CPU (Groq API) | `evals/run_api_arm.py` |
| Deterministic scoring (validity, acc, macro-F1) | local CPU / **CI** | `evals/score.py` |
| Reply quality (LLM-as-judge, gated) | local CPU (Groq API) | `evals/judge.py` |
| Report + charts | local CPU | `evals/report.py` |
| Quality gate | **CI** | `evals/check_thresholds.py` |

Raw model completions are **committed** under `artifacts/completions/`, so
every number in this README can be re-derived by anyone with
`python evals/score.py` — no GPU, no API keys. GitHub Actions re-runs the
scoring + quality gate on every push that touches prompts, parsing, or
artifacts.

## Reproduce

```bash
# local (CPU)
pip install -r requirements.txt
cp .env.example .env                       # add GROQ_API_KEY
python data/prepare_data.py --verify-against <path-to-support-ticket-classifier>/data
python data/build_sft_dataset.py
python data/make_eval_subsets.py
python evals/run_api_arm.py                # Groq 70B reference arm (~2h paced, cached)

# GPU (free Kaggle/Colab T4)
# notebooks/01_train_and_push.ipynb  — train + push to HF Hub
# notebooks/02_serve_bench_demo.ipynb — vLLM serve + completions + benchmark

# scoring (CPU, no GPU/keys needed)
python evals/score.py && python evals/judge.py && python evals/report.py
python evals/check_thresholds.py
```

## Design notes

- **Why a 4B model?** Qwen3-4B-Instruct-2507 is the largest open model
  (Apache-2.0) that both QLoRA-trains and vLLM-serves on a free 16 GB T4
  (fp16 only — sm75 has no bf16).
- **Why commit completions?** Eval claims rot when they can't be re-checked.
  Committing raw outputs makes the comparison auditable and lets CI act as a
  regression gate for prompt/parser changes.
- **Why is the Groq arm only 550 rows?** Free-tier TPM budget. It's a
  stratified 50/class sample reported with Wilson 95% CIs; the base/tuned arms
  use the full 2,464-row test set.
